问题与动机

1.1 背景：要解决的任务

一个典型的视频创作流程包含多个步骤：从图像或文本构思出发，生成视频，再进行编辑、分割关键对象，最终合成完整场景。[Section 1, 第1段]

图像/文本构思 → 视频生成 → 视频编辑 / 分割关键对象 → 合成场景
传统方法的问题：完成这样的流程需要手动拼接多个独立工具（stitching together disparate tools），导致流程脆弱、劳动密集、缺乏交互性（brittle, labor-intensive, and non-interactive），无法提供自动化和主动辅助。[Section 1, 第1段]

1.2 现有方法的问题

原文指出现有方法的三类局限 [Section 1, 第2-3段]：

① 单任务视频模型（Single-task video models）

做法：为特定任务设计专用网络，如分割网络、视频生成网络等，在各自任务上达到高性能。

局限：孤立运行（operate in isolation），无法处理多步骤目标，需要人工协调才能组合使用。

② 视频-语言基础模型（Video-language foundation models）

代表工作：VILA-U [Wu et al., 2024]、Show-o2 [Xie et al., 2025]、Omni-video [Tan et al., 2025]、Vitron [Fei et al., 2024]

做法：尝试将理解和生成整合到单一模型中（integrate understanding and generation into one model），通过大规模多模态训练学习广泛的能力。

局限：整体式且不灵活（monolithic and inflexible），难以整合新工具或模块化功能，用于复杂工作流时效率低下或不切实际。

③ 基于 LLM 的工具使用 Agent（LLM-based agents with tool use）

代表工作：

  • HuggingGPT [Shen et al., 2023]：使用语言模型作为控制器来规划任务，并按顺序调用合适的模型/工具（employs a language model as a controller to plan tasks and invoke appropriate models/tools in sequence）
  • VideoAgent [Fan et al., 2024]：利用 LLM 配合结构化记忆和预定义的视频工具集来回答长视频问题（leverages an LLM with a structured memory and a predefined set of video tools to answer questions on long videos）
局限：HuggingGPT 是通用框架，未针对视频操作专门优化（not specialized for detailed video operations）；VideoAgent 主要关注视频理解问答（focuses mainly on video understanding queries），编辑/生成能力有限。
关键瓶颈：缺乏一个能够跨多个视频任务和步骤进行推理的统一系统，这已成为下一代视频智能的关键瓶颈（critical bottleneck）。目前没有任何平台能完整支持统一的、端到端的智能体视频工作流，无缝集成理解、生成、编辑，并具备主动交互和可扩展模块化能力。[Section 1, 第1、3段]

1.3 本文贡献

原文提出三项贡献 [Section 1, Contributions]：

(1) UniVA 框架：首个开源、统一、全能（open-source, unified, omni-capable）的视频通才框架，将视频理解、编辑、生成统一到单一智能体工作流，基于 Plan-Act 双智能体架构和多层级记忆设计
(2) MCP 工具生态：基于 MCP 协议 [Anthropic, 2024] 的模块化视频生产引擎（modular, MCP-based video production engine），支持即插即用的开源或 API 工具，实现任意条件生成、跨模态编辑和工业级质量控制
(3) UniVA-Bench 基准：首个评估智能体视频智能（agentic video intelligence）的基准，不仅评估任务性能，还评估规划质量、记忆利用和工具路由效率

核心方法

3.1 UniVA 控制核心（The UniVA Control Core）

UniVA 的控制核心包含两个关键组件：负责任务规划的 决策引擎（decision engine），以及管理中间状态和交互历史的 记忆系统（memory system）。[Section 3.2, 第1段]

3.1.1 Plan-Act 双智能体架构（Plan–Act Dual Agent Architecture）

Figure 2: UniVA 整体架构 - Plan-Act 双智能体范式

架构设计：UniVA 的核心是 Plan-Act dual-agent architecture。策略 π 通过双智能体 Plan-Act 框架实现。[Section 3.2.1, 第1段]

组件 角色 职责 出处
Plan Agent（规划智能体） high-level policy network（高层策略网络） 将目标 G 和当前状态 sₜ 映射为抽象的 plan 序列 [Section 3.2.1, 第1段]
Act Agent（执行智能体） low-level execution policy（低层执行策略） 将 plan 的每个步骤转换为具体动作 aₜ，通过 MCP 接口选择合适工具并填充参数 [Section 3.2.1, 第1-2段]

设计原理：这种分离使 Planner 保持 lightweight and strategic（轻量级和战略性），而 Actor 专注于 reliable and efficient tool use（可靠高效的工具使用）。[Section 3.2.1, 第2段末]

Re-planning 机制

原文提到这种 planning 和 acting 的分离允许系统 "look ahead and reason about long-horizon goals, while flexibly adapting the plan if intermediate results require changes"（前瞻性地推理长程目标，同时在中间结果需要变化时灵活调整计划）。[Section 1, 第1项第1点]

Plan Agent 提示词（Appendix A.3）中对 re-planning 的规则定义：

"If a step completes with a failure status, the planning should be dynamically adjusted."
[Appendix A.3, IV. Planning Updates]

论文提出 ReplanQ（Re-planning Quality） 指标来衡量系统从失败中恢复的效率。[Section 4.2]

# Unified Video Planner Agent

## Role
You are Univideo, an expert video generation
and processing planner. Your task is to
analyze user requests and create detailed,
step-by-step execution plans using available
tools. You must break down complex tasks into
manageable steps and select appropriate tools
for each operation.
▸ [Role] 角色定义

定义 Agent 身份为"视频规划专家"，核心职责是任务分解和工具选择。通过明确的角色设定约束 LLM 的输出行为。

## Available Tools Overview
The tools are divided to **atom** and **workflow**:
- **Atom**: Basic, independent functional modules,
  low-level operations
- **Workflow**: Series of operations with clear
  steps and sequence

### Video Generation Tools:
#### Atom Functions:
- text2video_gen: ~5s video from text
- image2video_gen: ~5s video from text + image
- video_extension: Extend video from last frame
- frame2frame_video_gen: Transition between frames

#### Workflow Functions:
- storyvideo_gen: Full story pipeline
- entity2video: Story video with character images
▸ [I. AVAILABLE TOOLS] 工具清单

Atom vs Workflow 分层设计：原子工具是最小执行单元，工作流则封装了多步骤流程。这种分层让 Plan Agent 可以灵活组合原子操作，也可直接调用预定义工作流简化规划。

storyvideo_gen 等 Workflow 内部自动处理：Storyboard → 角色生成 → 关键帧 → 片段合并，Plan Agent 无需手动编排这些步骤。

### Video Editing Tools (Atom):
- depth_modify: Edit foreground/background using depth
- style_transfer: Convert video to artistic style
- repainting: Repaint/replace specific objects

### Video Understanding Tools:
- vision2text_gen (Atom): Visual → Text description
- video_timestamp_analysis (Workflow): Frame analysis
- main_object_analysis (Workflow): Object analysis

### Video Tracking Tools:
- video_referring_segmentation: Instance segmentation

### Image Generation Tools:
- text2image_generate: Image from text
- image2image_generate: Image with style consistency
▸ 工具分类（续）

工具覆盖视频全生命周期：生成 → 编辑 → 理解 → 跟踪。每个工具都明确说明输入输出，便于 Plan Agent 构建依赖链。

vision2text_gen 是关键的"桥接工具"——当用户只提供视觉素材而无文字描述时，先调用它理解内容再规划后续步骤。

## II. Core Planning Logic & Tool Selection

### Basic Calling Logic:
- If user does not provide semantic information
  about visual content → use vision2text_gen first
- For materials not provided by user → generate
  using existing tools first, then proceed
- Style consistency requires careful reference
  management
▸ [II. CORE PLANNING LOGIC] 规划逻辑

三条核心规则指导工具选择：

① 信息补全原则：缺语义信息先理解
② 素材生成原则：缺素材先生成
③ 一致性原则：跨步骤保持风格

## III. Plan Output Format
{
  "task_analysis": "Brief description of task type",
  "execution_plan": {
    "total_steps": N,
    "steps": [{
      "step_number": 1,
      "action_description": "What this step does",
      "tool": {
        "name": "tool_name",
        "purpose": "What this achieves",
        "input_requirements": ["material 1", ...]
      },
      "dependencies": [],
      "status": "ongoing/pending/success/failure",
      "output": ""
    }]
  }
}
▸ [III. PLAN OUTPUT FORMAT] 输出格式

严格的 JSON Schema 约束输出结构，每个字段都有明确语义：

• dependencies：显式建模步骤依赖关系
• status：支持状态机转换
• input_requirements：声明每步所需输入，便于 Act Agent 执行

⚠️ 提示词强调：每次只能输入一张图像，多素材需分步调用 

## IV. Planning Updates
- At beginning: first step = ongoing, rest = pending
- After each step: update status based on result
- Only sequential execution allowed
- Only ONE step can be ongoing at any time
- If step fails → dynamically adjust planning
- Each plan output must have one ONGOING step
- When plan ends successfully → output brief summary
▸ [IV. PLANNING UPDATES] 动态更新机制

实现 Plan-Act 循环的核心约束：

• 单步执行：避免并行带来的状态混乱
• 失败重规划：Re-planning 触发条件
• Ongoing 指示器：Act Agent 通过查找 status="ongoing" 确定当前执行步骤

✓ 这套状态机设计保证了 Plan-Act 解耦执行的确定性

关键设计要点
  • 工具分 Atom/Workflow：降低规划复杂度，支持灵活组合或快捷调用
  • dependencies 显式依赖：支持 DAG 结构的任务依赖建模
  • status 状态机：pending → ongoing → success/failure，驱动 Plan-Act 循环
  • 结构化 JSON 输出：便于程序解析，减少 Act Agent 理解成本

  • 信息流：Plan → Act（当前步骤）→ Act 执行 → Plan（更新状态）→ 循环

3.1.2 记忆机制（Memory Mechanism）

原文指出，agentic video systems 的一个关键挑战是 "maintain context across long and multi-step workflows"（跨长程多步骤工作流保持上下文）。UniVA 通过 three-level memory mechanism（三层记忆机制）来解决这一问题，与 Planner-Actor 循环配合工作。[Section 3.2.2, 第1段]

Figure 3: Memory-augmented framework（记忆增强框架）

记忆层 存储内容 作用（原文表述） 出处
Global Memory（全局记忆） persistent knowledge and reusable resources（持久知识和可重用资源），如 precomputed embeddings、generic video facts、tool usage statistics provides background context and supports cross-task generalization [Section 3.2.2, 第2段]
Task Memory（任务记忆） intermediate artifacts, tool outputs, and execution traces（中间产物、工具输出、执行轨迹） ensures continuity across multiple steps，允许后续 sub-goals 复用结果而无需重新计算；also enables traceability [Section 3.2.2, 第3段]
User Memory（用户记忆） user-specific preferences and historical interactions（用户特定偏好和历史交互），如 favored styles、recurring edit patterns enables adaptive behaviors，例如自动应用用户偏好的分辨率或编辑风格 [Section 3.2.2, 第4段]

整体设计：Global Memory 和 User Memory 共同构成长期历史 Hₜ 的持久存储，为策略 π 提供丰富上下文。Task Memory 维护当前任务的动态状态 sₜ 及其中间产物。[Section 3.2.2, 第5段]

def inject_execution_results(self, session_id, plan, execution_results: Dict[int, Any]) -> Dict:
    """将执行结果注入记忆，供下一轮规划使用"""
    try:
        current_state = self.agent.get_session_state(session_id).get("execution_history", [])
    except Exception:
        current_state = []

    current_state.append({
        "plan": plan,
        "execution_results": execution_results
    })

    self.agent.update_session_state(
        session_state_updates={"execution_history": current_state},
        session_id=session_id
    )

5.4.3 智能体指标（Agentic Metrics）

原文 Section 4.3 和 Appendix B.2 定义了三个专门评估规划能力的指标 [Appendix B.2, Agentic metrics 段]：

① wPED（Weighted Plan Edit Distance）

一句话解释：衡量 Agent 的规划和专家规划有多像。

通俗理解：假设专家的做法是"先切菜→再炒菜→最后装盘"，如果 Agent 的做法是"先切菜→再装盘→最后炒菜"，就需要"交换两步"才能变成专家做法。需要改的步骤越少，wPED 分数越高。

② DepCov（Dependency Coverage）

一句话解释：衡量 Agent 是否遵守"先后顺序"的规则。

通俗理解：有些事情有固定的先后顺序，比如"必须先生成视频才能编辑视频"、"必须先分割出人物才能换背景"。DepCov 检查 Agent 的计划是否违反了这些常识性的顺序规则。违反越少，分数越高。

③ ReplanQ（Re-planning Quality）

一句话解释：衡量 Agent 出错后能否"小修小补"而不是"推倒重来"。

通俗理解：假设 Agent 执行到第 3 步时失败了，好的 Agent 应该只调整第 3 步及之后的计划，而不是把整个计划全部重写。修改越少、恢复越快，ReplanQ 分数越高。

06

结论与局限性

6.1 核心结论

论文认为视频领域的进步需要范式转变：从开发孤立的单任务模型转向创建能够处理复杂协作工作流的集成系统。[Section 5, 第1段]

主要贡献：
  • 开发了功能强大且可扩展的 UniVA 平台
  • 展示了其作为全能视频通才的涌现协同能力（emergent synergistic capabilities）
  • 发布 UniVA-Bench 以严格衡量此类进展

6.2 关键发现

发现一：广度验证

实验验证了 UniVA 的广度（breadth）：在各类视频任务上展现出有竞争力的性能。[Section 5, 第2段]

发现二：深度验证 — Agentic Synergy

更重要的是，论文揭示了 UniVA 的深度（depth）：通过 "Agentic Synergy"（智能体协同），即工具之间信息流的动态管理，UniVA 能够解决孤立模型难以处理的复杂一致性问题。[Section 5, 第2段]

核心洞见："This confirms that UniVA is not merely a collection of tools, but an engine for generating emergent intelligence."
（这证实了 UniVA 不仅仅是工具的集合，而是一个产生涌现智能的引擎。）[Section 5, 第2段]

6.3 局限性

DINO Score 权衡

在 Entities2Video 任务中，专用模型（如 Seedance）在主体一致性（DINO Score）上表现更强，而 UniVA 在这方面仍有竞争力但非最优。[Section 4.3.1, Entities2Video 段]

论文原文："This highlights a current trade-off where our agent prioritizes overall instruction complexity and narrative coherence, a direction for future optimization."
（这表明当前存在一个权衡：我们的 agent 优先考虑整体指令复杂性和叙事连贯性，这是未来优化的方向。）[Section 4.3.1]

具体表现：在 Entities2Video 任务中，UniVA 的 DINO Score（0.8796）略低于 Seedance（0.8800），但 CLIP Score 和 MLLM Judge 表现良好，说明系统更侧重于指令遵循而非帧级相似度。

个人思考与启发

8.1 对我们研究的启发

Plan 时输出依赖关系

在规划阶段需要显式输出步骤间的依赖关系，这样可以：

  • 并行处理：无依赖的步骤可以同时执行，提升效率
  • 错误时及时调整策略：当某步骤失败时，能够准确判断哪些后续步骤受影响，避免无效执行
Case: 披萨

Re-planning 机制

Re-planning 是必要机制，不仅用于出错纠正，更重要的是：

人在看到一个 Question 后的回答策略，在看到字幕内容或关键帧内容后会发生很大的改变。初始规划往往基于对问题的理解，但实际执行中获取的信息可能完全改变最优策略。

Case: Lunar

Workflow 预定义

为不同类型的任务预定义部分处理步骤，因为针对整体的问题和针对片段的问题，处理步骤是不一样的：

问题类型 Workflow
定位 + 回答问题 先时间定位 → 再针对片段回答
排序问题 先识别所有相关事件 → 再确定时序关系

预定义 workflow 可以减少 Agent 的决策负担，提高执行效率和准确性。

Case: correct order
